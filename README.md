# คู่มือเลือกโมเดล Claude ให้เต็มประสิทธิภาพ

> เลือกโมเดลตาม **"ความเสี่ยง/ความยากของงาน"** ไม่ใช่ตามฟีเจอร์
> ทุ่มความฉลาด (แพง) ตรงที่พลาดแล้วเจ็บ · ประหยัดตรงที่พลาดแล้วแก้ง่าย

---

## สรุปเร็ว — โมเดลไหนทำอะไร

| โมเดล | บทบาท | ใช้กับ |
|---|---|---|
| **Opus 4.8** | สมอง | วางโครง, ตัดสินใจ, การเงิน, รอยต่อระบบ, debug ยาก, ออกแบบ UI/UX |
| **Sonnet 5** | มือ | เขียนโค้ดทั่วไป, component, API, แปลง Figma→code (คุ้มสุด) |
| **Haiku 4.5** | แรงงาน | งานถึกซ้ำๆ: rename, format, test, commit, boilerplate |
| **Fable 5** | ปาก | คำบนหน้าจอ, copy การตลาด, docs เชิงเล่าเรื่อง |

> 3 ตัวแรกวัดกันบนแกน **ฉลาด vs เร็ว vs ถูก** · Fable อยู่คนละแกน = งานภาษาสำหรับคนอ่าน

---

## หลักคิดหลัก

```
ตัดสินใจ = Opus   |   ลงมือ = Sonnet   |   งานถึก = Haiku   |   คำพูด = Fable
```

**แพทเทิร์นทุกงาน:**
1. Opus → วางแผน + ออกแบบ (ครั้งเดียว)
2. Sonnet → ลงมือเขียน (ส่วนใหญ่ของงาน)
3. Haiku → subagent ทำงานย่อยขนานกัน
4. Opus → กลับมาตอน "ติด" หรือ review ตอนจบ

---

## แบ่งตามสาย

### Frontend
- ออกแบบ UI/UX, component architecture → **Opus**
- เขียน component, Figma→code, styling → **Sonnet**
- แก้ CSS เล็ก, rename props → **Haiku**
- debug state/render งง → **Opus**

### Backend
- ออกแบบ API/schema/data model → **Opus**
- endpoint, business logic, CRUD → **Sonnet**
- validation, error handling → **Sonnet** (ห้าม Haiku)
- test, boilerplate, DTO → **Haiku**
- debug race/N+1, optimize query → **Opus**

### DevOps
- ออกแบบ CI/CD, infra architecture → **Opus**
- Dockerfile, CI yaml → **Sonnet**
- แก้ config, env var, bump version → **Haiku**
- debug pipeline พัง (cert/network/permission) → **Opus**
- Terraform/K8s → **Opus ออกแบบ → Sonnet ลงมือ**

---

## ระบบใหญ่โยงกัน (แชต + การเงิน + สมาชิก)

**แบ่งตามความเสี่ยง:**

| ส่วน | ความเสี่ยง | โมเดล |
|---|---|---|
| 💰 การเงิน (ยอด, ธุรกรรม, refund) | สูงสุด | **Opus ทั้งหมด** |
| 👤 สมาชิก (auth, สิทธิ์, tier) | สูง | **Opus ออกแบบ / Sonnet เขียน** |
| 💬 แชต (ส่งข้อความ, realtime) | กลาง | **Sonnet เป็นหลัก** |
| 🔗 รอยต่อ (แชตหักเงิน, tier กำหนดสิทธิ์) | สูงสุด | **Opus เสมอ** |

**กฎเหล็ก:**
1. แตะเงิน = Opus เท่านั้น (idempotency, race, double-spend คือกับดักที่ดู "ง่าย")
2. auth/สิทธิ์ = อย่างน้อย Sonnet, ห้าม Haiku
3. รอยต่อระหว่างระบบ = Opus (บั๊กร้ายซ่อนตรงนี้)
4. แชตล้วน + งานซ้ำ = Sonnet/Haiku
5. ออกแบบ contract (Phase 0) ก่อนเขียนเสมอ ด้วย Opus

**Phase 0 (Opus ทำครั้งเดียว):** ใครเป็นเจ้าของ data อะไร · sync vs async · ถ้าล้มกลางทางทำยังไง (rollback/compensation)

---

## วางโครงระบบ → Opus (คุ้มค่าจ่ายแพงสุด)

- โครงผิด แก้ทีหลังแพงกว่าโค้ดผิด 100 เท่า
- ให้ Opus **เสนอ 2-3 แนวทาง + trade-off** แล้วเลือก ไม่ใช่คิดรอบเดียวจบ
- ⚠️ สั่ง "ให้พอดีสเกลตอนนี้ ไม่ over-engineer" — โมเดลฉลาดชอบออกแบบเกินจำเป็น

---

## ทำ UI ให้สวย

**เก่งสุด = Opus** แต่ความจริง: **"สวย" ไม่ได้อยู่ที่โมเดล อยู่ที่เห็นของจริง**

```
เขียน UI → เปิดดูจริง (preview) → จับจุดไม่สวย → แก้ → ดูใหม่  (วนจนสวย)
```

โมเดลกลางๆ + เห็นผลลัพธ์ + วนแก้ 3 รอบ > โมเดลเทพที่เขียนรอบเดียวไม่เคยเห็นหน้าจอ

**เครื่องมือสำคัญกว่าอัปเกรดโมเดล:**
- **Figma MCP** — มีดีไซน์แล้ว ดึงมาเป็นโค้ดตรงๆ
- **Preview tools** — เปิดหน้าจริง วัดสี/ระยะ ให้แก้จากที่เห็น
- **Artifact** — สร้าง mockup ดูในแชตทันที

**กุญแจ:** ให้ตัวอย่างอ้างอิง (ลิงก์เว็บที่ชอบ) + บอกสไตล์ชัด ดีกว่าบอก "ทำให้สวย" ลอยๆ

---

## Fable 5 เอาไว้เขียนอะไร

**คำพูดบนหน้าจอ:** ปุ่ม, empty state, error message, loading, onboarding, confirmation
**Copy การตลาด:** landing headline, CTA, email subject, push notification
**Docs เล่าเรื่อง:** README อ่านสนุก, blog, changelog, mock data สมจริง

**เส้นแบ่ง:** ข้อความที่ *คนอ่าน* = Fable · โค้ดที่ *เครื่องรัน* = ไม่ใช่ Fable
**ลำดับถูกต้อง:** Opus คิด → Sonnet ทำ → **Fable เกลาคำตอนท้าย** (ไม่ใช่ให้ Fable นำหน้า)

### วิธีเขียน copy การตลาดให้ดี
ป้อนข้อมูล 5 อย่าง: **ขายอะไร / ให้ใคร / แก้ปัญหาอะไร / โทนไหน / อยากให้ทำอะไร (CTA)**
โครง: `Hook → Value → Proof → CTA` · พูดถึงลูกค้าไม่ใช่ตัวเอง · ขอหลายเวอร์ชันแล้วเลือก

---

## git commit → Haiku (หรือตัวที่เพิ่งเขียนโค้ด)

- commit ปกติ = **Haiku** (แค่บรรยาย diff)
- ตัวที่เพิ่งเขียนโค้ด commit เองเลย (มี context อยู่แล้ว ไม่ต้องส่งต่อ)
- commit ที่ต้องอธิบาย "ทำไม" (การเงิน/security) = **Sonnet**

## มุมมองซีเนียร์ — ตัดกฎทิ้งเกือบหมด

ตารางทั้งหมดข้างบนคือของ "มือใหม่" ที่ต้องมีกฎกำกับ ซีเนียร์จริงมี default แล้วสลับเฉพาะตอนมีเหตุผล:

```
default: Sonnet 5 ทุกอย่าง
  ↑ ขึ้น Opus เมื่อ "รู้สึกว่ามันยาก" หรือ "พลาดแล้วเจ็บ"
  ↓ ลง Haiku เมื่อ "งานนี้ไม่ต้องคิดเลย" และมีเยอะพอจะคุ้ม
```

**ความต่าง มือใหม่ vs ซีเนียร์:**

| | มือใหม่ | ซีเนียร์ |
|---|---|---|
| เริ่มงาน | เปิดตารางหาว่าใช้ตัวไหน | เปิด Sonnet ลุยเลย |
| เจองานยาก | ลังเล ลองตัวถูกก่อน | ขึ้น Opus ทันที ไม่เสียดาย |
| งานถึก | ใช้ Opus เพราะเคยชิน | เด้งลง Haiku / มอบ subagent |
| สลับโมเดล | บ่อย ตามฟีเจอร์ | นานๆ ที ตามสัญชาตญาณ |

**กฎเดียวที่ไม่เคยยอมประหยัด:** "พลาดแล้วเจ็บ" (เงิน, auth, migration ลบข้อมูล, decision แก้ทีหลังแพง) = ใช้ตัวที่ดีที่สุด ไม่ต้องคิด · ที่เหลือแก้ง่ายก็ใช้ตัวถูก ผิดก็แก้

**ท่าจริงใน Claude Code:** ตัวหลัก (Opus/Sonnet) คุมงาน → มอบงานย่อยให้ subagent (ตัวถูก) ทำขนาน — คนคุมฉลาด งานถึกให้ลูกน้องถูกๆ ทำ

> **กฎทอง:** อย่าเลือกโมเดลให้เยอะกว่าเลือกโค้ด
> เวลาที่เสียไปนั่งเลือกโมเดลเป๊ะๆ ทุกงาน แพงกว่าส่วนต่างค่าโมเดลที่ประหยัดได้

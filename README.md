# เลือกโมเดล Claude ให้เต็มประสิทธิภาพ

> **หลักเดียวที่ต้องจำ:** เลือกโมเดลตาม *ความเสี่ยงของงาน* ไม่ใช่ตามฟีเจอร์
> ทุ่มความฉลาดตรงที่พลาดแล้วเจ็บ — ประหยัดตรงที่พลาดแล้วแก้ง่าย

---

## 1. รู้จักทีมของคุณ

| โมเดล | บทบาท | ถนัด |
|---|---|---|
| **Opus 4.8** | 🧠 สมอง | คิด ตัดสินใจ ออกแบบ แก้ปัญหายาก |
| **Sonnet 5** | ✋ มือ | ลงมือเขียนโค้ด — ค่า default ที่คุ้มสุด |
| **Haiku 4.5** | 💪 แรงงาน | งานถึกซ้ำๆ ที่ไม่ต้องคิด ทำขนานได้เยอะ |
| **Fable 5** | 💬 ปาก | ภาษาสำหรับคนอ่าน — copy, microcopy, เล่าเรื่อง |

สามตัวแรกอยู่บนแกนเดียวกัน: *ฉลาด ↔ เร็ว ↔ ถูก*
Fable อยู่คนละแกน: งานของมันคือ **คำ** ไม่ใช่ **โค้ด**

**ลำดับที่ถูกต้องเสมอ:**

```
Opus คิด  →  Sonnet ลงมือ  →  Haiku เก็บงานถึก  →  Fable เกลาคำตอนท้าย
```

อย่าให้ Fable นำหน้า — มันเขียนเพราะ แต่คิดไม่ลึก
โครงต้องมาจาก Opus ก่อน แล้ว Fable ค่อยแต่งหน้าให้สวย

## 2. สูตรสำเร็จทุกโปรเจกต์

```
1. Opus    วางแผน + ออกแบบ            (ครั้งเดียว — จ่ายแพงตรงนี้คุ้มสุด)
2. Sonnet  ลงมือเขียน                  (ส่วนใหญ่ของงาน)
3. Haiku   subagent เก็บงานย่อยขนานกัน  (test, boilerplate, format)
4. Opus    กลับมาตอนติด + review ก่อนจบ
```

**กฎเหล็ก 3 ข้อ — ไม่มีข้อยกเว้น:**

1. **แตะเงิน = Opus** ต่อให้ดูง่ายแค่ไหน (idempotency, race, double-spend คือกับดักที่ *ดูง่าย*)
2. **auth / สิทธิ์ = อย่างน้อย Sonnet** ห้าม Haiku — validation พลาดจุดเดียว ประตูเปิดทั้งบ้าน
3. **รอยต่อระหว่างระบบ = Opus** บั๊กร้ายแรงไม่ได้อยู่ *ใน* ระบบ แต่อยู่ *ระหว่าง* ระบบ

## 4. เคสจริง

### 🌐 Landing Page — งานเบา ประหยัดได้เต็มที่

```
Opus    วาง layout + สไตล์ (ครั้งเดียว)
Sonnet  เขียน HTML/CSS จริง + วนแก้กับ Preview
Fable   headline, CTA, คำโปรย
Haiku   responsive, SEO meta, เก็บรายละเอียด
```

**ท่าเร็วสุด:** Opus ทำ mockup ใน Artifact → วนแก้จนชอบ → Sonnet ขยายเป็นโปรเจกต์จริง → Fable เกลาคำ
**กุญแจ:** ส่งลิงก์เพจที่ชอบมาเป็นตัวอย่าง + บอกกลุ่มลูกค้า — ดีกว่าบอก "ทำให้สวย" ลอยๆ สิบเท่า

### 🏪 ERP / POS — งานเสี่ยงสูง แบ่งตามความเสี่ยง

| โมดูล | ความเสี่ยง | โมเดล |
|---|---|---|
| 💰 ชำระเงิน, บิล, ทอนเงิน, ปิดยอด | สูงสุด | **Opus ทั้งหมด** |
| 📦 สต็อก, ตัดสต็อก, รับเข้า | สูงสุด | **Opus** |
| 🧾 บัญชี, ledger, ภาษี | สูงสุด | **Opus** |
| 👤 พนักงาน, สิทธิ์, กะ | สูง | Opus ออกแบบ / Sonnet เขียน |
| 🛒 หน้าขาย, ตะกร้า, บาร์โค้ด | กลาง | Sonnet |
| 📊 dashboard, รายงาน, CRUD สินค้า | ต่ำ–กลาง | Sonnet |
| 🎨 ฟอร์ม, template ใบเสร็จ | ต่ำ | Haiku |

**กับดัก POS ที่ดู "ง่าย" แต่ฆ่าร้าน — ต้อง Opus:**
- ตัดสต็อก + บันทึกขาย ต้อง **atomic** (ขายแล้วสต็อกไม่ลด = ของหาย)
- สองแคชเชียร์ขายชิ้นสุดท้ายพร้อมกัน → สต็อกติดลบ
- **Offline mode** — เน็ตหลุดต้องขายต่อได้ แล้ว sync ไม่ให้ยอดชน (ยากสุดใน POS)
- ปิดยอดสิ้นวันต้องตรงเป๊ะกับบิลทุกใบ
- เงิน คำนวณเป็น integer (สตางค์) — **ห้าม float**

**ลำดับ:**
```
Phase 0  Opus   contract ขาย/สต็อก/บัญชี + แผน offline (ตัดสินใจตั้งแต่ต้น)
Phase 1  Opus   core เงิน + ตัดสต็อก        │ Sonnet  POS UI + CRUD (ขนาน)
Phase 2  Opus   เชื่อม ขาย×สต็อก×บัญชี + ปิดยอด
Phase 3  Haiku  ใบเสร็จ ฟอร์ม รายงาน test data
Review   Opus   ไล่ path เงิน+สต็อก ก่อน go-live
```

### 🔗 ระบบใหญ่โยงกัน (แชต + การเงิน + สมาชิก)

หลักเดียวกับ POS: **การเงิน = Opus ทั้งหมด · สมาชิก = Opus ออกแบบ/Sonnet เขียน · แชต = Sonnet · รอยต่อ = Opus เสมอ**

ก่อนเขียนโค้ด ให้ Opus ทำ **Phase 0** ครั้งเดียว:
1. ใครเป็นเจ้าของ data อะไร (ยอดเงินอยู่ที่ระบบการเงินเท่านั้น — ห้ามเก็บสำเนา)
2. คุยกันแบบไหน (หักเงิน = เรียกตรงรอผล / แจ้งแบน = ยิง event)
3. ล้มกลางทางแล้วทำยังไง (rollback / compensating transaction)

## 5. เรื่องที่คนเข้าใจผิดบ่อย

### "ทำ UI ให้สวย" — ความสวยไม่ได้อยู่ที่โมเดล

```
เขียน UI → เปิดดูจริง → จับจุดไม่สวย → แก้ → ดูใหม่   (วนจนสวย)
```

โมเดลกลางๆ ที่ *เห็นผลลัพธ์* แล้ววนแก้ 3 รอบ **ชนะ** โมเดลเทพที่เขียนรอบเดียวโดยไม่เคยเห็นหน้าจอ

เครื่องมือสำคัญกว่าอัปเกรดโมเดล:
- **Figma MCP** — มีดีไซน์แล้ว ดึงมาเป็นโค้ดตรงๆ ไม่ต้องเดา
- **Artifact / Preview** — เห็นของจริง วัดสี วัดระยะ แก้จากที่เห็น
- **ตัวอย่างอ้างอิง** — ลิงก์เว็บที่ชอบ 1 อัน มีค่ากว่าคำสั่ง "สวยๆ" ร้อยคำ

### วางโครงระบบ — จุดที่คุ้มค่าจ่ายแพงสุด

โครงผิด แก้ทีหลังแพงกว่าโค้ดผิด 100 เท่า → **Opus เท่านั้น** และอย่าให้คิดรอบเดียวจบ:

```
เสนอ 2–3 แนวทาง → เทียบ trade-off กับข้อจำกัดจริง → เลือก + เขียนเหตุผลว่าทำไมไม่เลือกอันอื่น
```

⚠️ กำกับด้วย: *"ให้พอดีสเกลตอนนี้ ไม่ over-engineer"* — โมเดลฉลาดชอบออกแบบเผื่ออนาคตเกินจริง

### Fable ใช้ตรงไหน

| ✅ ใช่ | ❌ ไม่ใช่ |
|---|---|
| ข้อความที่ *คนอ่าน* | โค้ดที่ *เครื่องรัน* |
| ปุ่ม, error message, empty state, onboarding | API docs ที่ต้องเป๊ะ → Sonnet/Opus |
| headline, CTA, email, push notification | logic, โครงสร้าง → Opus |
| README เล่าเรื่อง, blog, changelog | comment ในโค้ด → ตัวที่เขียนโค้ด |

**สูตร copy การตลาด:** บอก 5 อย่าง — ขายอะไร / ให้ใคร / แก้ปัญหาอะไร / โทนไหน / CTA อะไร
โครง `Hook → Value → Proof → CTA` · พูดถึง *ลูกค้า* ไม่ใช่ตัวเอง · ขอ 3–5 เวอร์ชันแล้วเลือก

### git commit

Commit = งานถึก → **Haiku** พอ
แต่ถ้าตัวที่เพิ่งเขียนโค้ดยังถือ context อยู่ ให้มัน commit เองเลย — สลับไป Haiku เพื่อประหยัดมัก *ไม่คุ้ม* เพราะต้องอ่าน diff ใหม่
Commit ที่ต้องอธิบาย "ทำไม" (เงิน/security) → Sonnet

## 6. มุมมองซีเนียร์ — ตัดกฎทิ้งเกือบหมด

ตารางทั้งหมดข้างบนคือของมือใหม่ที่ต้องมีกฎกำกับ ซีเนียร์คิดแค่นี้:

```
default: Sonnet 5 ทุกอย่าง
  ↑ ขึ้น Opus   เมื่อ "รู้สึกว่ายาก" หรือ "พลาดแล้วเจ็บ"
  ↓ ลง Haiku   เมื่อ "ไม่ต้องคิดเลย" และมีเยอะพอจะคุ้ม
```

| | มือใหม่ | ซีเนียร์ |
|---|---|---|
| เริ่มงาน | เปิดตารางหาว่าใช้ตัวไหน | เปิด Sonnet ลุยเลย |
| เจองานยาก | ลังเล ลองตัวถูกก่อน | ขึ้น Opus ทันที ไม่เสียดาย |
| งานถึก | ใช้ Opus เพราะเคยชิน | เด้งลง Haiku / มอบ subagent |
| สลับโมเดล | บ่อย ตามฟีเจอร์ | นานๆ ที ตามสัญชาตญาณ |

กฎเดียวที่ซีเนียร์ไม่เคยยอมประหยัด: **"พลาดแล้วเจ็บ" = ใช้ตัวที่ดีที่สุด ไม่ต้องคิด**
(เงิน, auth, migration ลบข้อมูล, decision ที่แก้ทีหลังแพง)

ท่าจริงใน Claude Code: ตัวหลักฉลาดคุมงาน → มอบงานถึกให้ subagent ตัวถูกทำขนาน
เหมือนซีเนียร์จริงที่ไม่ลงมือ format code เอง

> **กฎทอง:** อย่าเลือกโมเดลให้เยอะกว่าเลือกโค้ด
> เวลาที่เสียไปนั่งเลือกโมเดลเป๊ะๆ ทุกงาน แพงกว่าส่วนต่างค่าโมเดลที่ประหยัดได้
